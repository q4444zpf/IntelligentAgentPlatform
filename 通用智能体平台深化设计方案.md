# 通用智能体平台深化设计方案

## 1. 目标与落地原则

本深化设计面向第一版可落地建设，目标是在 3-4 个月内交付一个可配置、可编排、可协同、可审计的通用智能体平台。

第一版不做“大而全”的开放生态，而是先形成稳定平台底座：

- 用户可以创建和维护自己的智能体、MCP、Skill。
- 用户可以将个人资源提交发布到公用资源库。
- 平台支持知识库问答、Tool 调用、流程编排和编排式多智能体协同。
- 管理员可以审核公用资源、配置大模型、查看审计日志。
- 第三方系统可以通过 OpenAPI 调用智能体和流程。

落地原则：

- 先做闭环，再做增强。
- 先做编排式协同，再做自治式协同。
- 先做 HTTP API / MCP / 知识库三类工具接入，再扩展插件市场。
- 所有资源必须有 owner、visibility、publish_status、version。
- 所有大模型调用、Tool 调用、流程节点执行、发布审核都必须留痕。
- Web 版智能体必须采用沙箱模式执行 Tool、MCP、脚本和文件处理任务，禁止用户自建能力直接访问平台服务器宿主环境。

### 2.1 Web 智能体沙箱架构

Web 版智能体面向多用户开放配置能力，用户可以维护自己的智能体、MCP 和 Skill。为防止服务器安全问题，所有非纯配置类执行必须进入沙箱层。

沙箱执行范围：

- MCP Tool 调用。
- 用户自建 Skill 调用。
- 文件解析、转换和内容提取。
- 脚本任务和代码执行。
- 第三方 API 代理调用。
- 流程节点中的 Tool / MCP 执行。

禁止行为：

- 普通用户配置的智能体直接执行服务器本地命令。
- 普通用户配置的 MCP 直接访问宿主机文件系统。
- 用户上传脚本在 API 服务进程内执行。
- Tool 执行过程读取平台配置文件、密钥文件或数据库连接信息。
- 未审核的公用资源访问内网敏感地址。

沙箱建议：

| 能力 | 第一版建议 |
|---|---|
| 执行隔离 | 独立容器或受限 Worker 进程 |
| 文件隔离 | 每次任务独立临时目录，任务结束清理 |
| 网络隔离 | 默认禁止访问内网网段，可配置域名白名单 |
| 资源限制 | CPU、内存、执行时长、并发数限制 |
| 权限限制 | 非 root 用户运行，禁止挂载宿主敏感目录 |
| 密钥注入 | 仅注入当前 Tool 所需临时凭证 |
| 日志记录 | 记录执行人、资源、参数摘要、耗时、退出状态 |

第一版可以先实现轻量沙箱：Tool Executor 将高风险任务投递到独立 Worker，Worker 在受限容器中执行；后续再演进为 Kubernetes Job 或专用沙箱集群。

## 3. 核心领域模型

### 3.1 资源统一模型

智能体、MCP、Skill、Prompt、知识库、流程都按统一资源模型管理。

| 字段 | 说明 |
|---|---|
| id | 主键 |
| resource_type | agent、mcp、skill、prompt、knowledge_base、workflow |
| name | 资源名称 |
| code | 资源唯一编码 |
| description | 描述 |
| owner_id | 创建人 |
| tenant_id | 租户 |
| space_type | personal、team、public、system |
| visibility | private、team、public |
| version | 当前版本 |
| publish_status | draft、private_active、pending_review、rejected、public_active、removed |
| runtime_status | enabled、disabled |
| created_at | 创建时间 |
| updated_at | 更新时间 |

统一资源模型用于解决三个问题：

- 个人资源、公用资源、系统资源的边界清晰。
- 发布审核和版本追溯可以复用。
- 权限判断可以在资源层统一处理。

### 3.2 智能体模型

智能体由基础信息、Prompt、模型、知识库、Skill、MCP、权限和安全策略组成。

关键表：

- `agent`
- `agent_version`
- `agent_prompt_binding`
- `agent_model_binding`
- `agent_skill_binding`
- `agent_mcp_binding`
- `agent_knowledge_base_binding`

智能体版本发布时，需要固化以下内容：

- Prompt 版本
- 大模型配置 ID
- 绑定 Skill 版本
- 绑定 MCP 版本
- 绑定知识库版本
- 输出格式
- 安全策略

这样历史会话可以追溯到当时使用的配置。

### 3.3 MCP 模型

MCP 作为一类可调用工具源管理。

关键表：

- `mcp_server`
- `mcp_server_version`
- `mcp_tool`
- `mcp_resource`
- `mcp_connection_check`
- `mcp_call_log`

第一版 MCP 支持：

- 远程 HTTP / SSE MCP。
- 本地或平台托管 MCP 的配置登记。
- 工具列表同步。
- 工具 Schema 查看。
- 绑定到智能体。
- 提交公用审核。

第一版不允许普通用户配置的 MCP 直接在平台 API 服务或宿主机执行。本地或平台托管 MCP 必须运行在沙箱容器或受限 Worker 中；远程 MCP 必须经过网络白名单、权限校验和调用审计。

### 3.4 Skill 模型

Skill 是平台原生工具，第一版优先支持 HTTP API 类型。

关键表：

- `skill`
- `skill_version`
- `skill_input_schema`
- `skill_output_schema`
- `skill_auth_config`
- `skill_call_log`

Skill 必须定义：

- 名称和描述。
- 输入 JSON Schema。
- 输出 JSON Schema。
- 调用方式。
- 鉴权方式。
- 超时和重试。
- 是否需要人工确认。
- 可见和可用范围。

### 3.5 流程模型

第一版流程编排只做基础可运行能力。

关键表：

- `workflow`
- `workflow_version`
- `workflow_node`
- `workflow_edge`
- `workflow_instance`
- `workflow_node_instance`
- `workflow_human_task`

节点类型：

- start
- agent
- skill
- mcp_tool
- condition
- human_confirm
- output
- end

第一版流程执行策略：

- 有向无环图优先。
- 暂不支持复杂循环。
- 条件节点支持简单表达式。
- 人工确认节点会暂停流程。
- 所有节点输入输出都落库。

### 3.6 多智能体协同模型

第一版采用“编排式协同”。

关键表：

- `agent_collaboration`
- `agent_collaboration_member`
- `agent_collaboration_version`
- `agent_collaboration_run`
- `agent_collaboration_step`

协同类型：

- sequential：顺序执行。
- master_worker：主从执行。
- summarize：多结果汇总。

第一版暂不支持智能体自主无限递归拉起其他智能体。

## 4. 核心运行链路

### 4.1 AI 对话链路

```text
用户发送消息
→ 鉴权与资源权限校验
→ 加载智能体发布版本
→ 加载会话历史和摘要
→ 注入页面上下文或附件上下文
→ 检索知识库
→ 组装 Prompt
→ 判断是否需要 Tool / MCP 调用
→ 通过沙箱层执行 Tool / MCP
→ 将工具结果回填上下文
→ 调用大模型生成回复
→ 流式返回结果
→ 记录消息、模型调用、工具调用、引用来源
```

### 4.2 Skill 调用链路

```text
智能体或流程节点请求调用 Skill
→ 校验用户权限、智能体权限、Skill 权限
→ 校验输入 Schema
→ 判断是否需要人工确认
→ 判断是否需要沙箱执行
→ 组装 HTTP 请求
→ 在受控沙箱或代理 Worker 中执行调用
→ 校验输出 Schema
→ 脱敏敏感字段
→ 返回调用结果
→ 记录调用日志
```

### 4.3 MCP 调用链路

```text
智能体选择 MCP Tool
→ 校验用户、智能体、MCP 权限
→ 获取 MCP Server 连接配置
→ 校验 Tool Schema
→ 判断高风险工具是否需要确认
→ 在沙箱层调用 MCP Tool
→ 获取工具结果
→ 裁剪和脱敏结果
→ 返回给智能体
→ 记录 MCP 调用日志
```

### 4.4 流程执行链路

```text
手动或 API 触发流程
→ 校验流程权限
→ 创建 workflow_instance
→ 加载 workflow_version
→ 执行 start 节点
→ 按 edge 推进节点
→ 每个节点创建 node_instance
→ 执行 agent / skill / mcp_tool / condition / human_confirm
→ 人工节点暂停并生成待办
→ 人工处理后继续执行
→ output 节点生成流程结果
→ 流程结束并记录状态
```

### 4.5 多智能体协同链路

```text
用户或流程触发协同任务
→ 加载协同配置版本
→ 创建 collaboration_run
→ 主智能体生成任务上下文
→ 子智能体按配置顺序执行
→ 每一步记录输入、输出、模型调用、Tool 调用
→ 汇总智能体生成最终结果
→ 必要时进入人工确认
→ 返回协同结果
```

## 5. 权限与发布设计

### 5.1 权限判断维度

每次访问资源时都按以下维度判断：

- 用户是否登录。
- 用户是否属于对应租户。
- 用户是否拥有菜单权限。
- 用户是否拥有资源可见权限。
- 用户是否拥有资源使用权限。
- 用户是否拥有发布、审核、下架权限。
- Tool / MCP 调用是否需要额外确认。

### 5.2 资源可见规则

| space_type | 可见范围 |
|---|---|
| personal | owner 可见，授权成员可见 |
| team | 团队成员可见 |
| public | 有平台访问权限的用户可见 |
| system | 系统内置，按角色可见 |

### 5.3 发布审核流程

```text
用户提交发布
→ 生成待审核记录
→ 系统自动校验
→ 审核人员查看配置、说明、权限、安全项和调试结果
→ 审核通过：生成 public 版本
→ 审核驳回：填写驳回原因
→ 用户修改后可重新提交
```

自动校验项：

- 名称、描述、使用说明完整。
- 至少有一次成功调试记录。
- 不包含明文密钥。
- 高风险工具已配置确认。
- 输入输出 Schema 合法。
- 资源引用均可访问。

### 5.4 公用资源使用规则

公用资源支持两种使用方式：

- 直接使用：用户引用公用智能体、MCP、Skill。
- 复制使用：用户复制一份到个人空间后修改。

直接使用时，资源更新由原作者或管理员维护。复制使用时，后续不自动跟随原资源更新。

## 9. AI 服务落地设计

### 9.1 Agent Runtime

Agent Runtime 负责执行一次智能体调用。

核心职责：

- 加载智能体版本配置。
- 组装 Prompt。
- 检索知识库。
- 判断和执行 Tool / MCP。
- 调用大模型。
- 返回流式结果。
- 写入全链路日志。

### 9.2 Tool Executor

Tool Executor 统一执行 Skill 和 MCP Tool，并负责把 Web 版智能体的高风险执行转交给 Sandbox Executor。

职责：

- 权限校验。
- Schema 校验。
- 人工确认。
- 超时和重试。
- 调用日志。
- 结果脱敏。

### 9.3 Sandbox Executor

Sandbox Executor 是 Web 版智能体的安全执行层。

职责：

- 为每次 Tool / MCP / 脚本 / 文件处理任务创建隔离执行环境。
- 限制 CPU、内存、磁盘、执行时长和并发。
- 限制网络访问范围，默认禁止访问内网地址。
- 提供临时工作目录，任务结束后清理。
- 只注入任务所需的最小权限凭证。
- 捕获 stdout、stderr、退出码和异常。
- 将执行结果返回给 Tool Executor。
- 写入沙箱执行日志。

### 9.4 LLM Router

LLM Router 负责选择和调用模型。

第一版支持：

- 默认模型。
- 智能体绑定模型。
- Embedding 模型。
- 调用失败记录。
- token 和耗时统计。

第二版再做复杂路由、备用模型和成本预算。

## 11. MVP 验收用例

### 11.1 个人资源与公用发布

- 普通用户创建个人智能体。
- 普通用户创建个人 HTTP Skill。
- 普通用户创建个人 MCP。
- 普通用户提交智能体、Skill、MCP 到公用。
- 审核人员审核通过。
- 其他用户在公用资源库看到资源。
- 其他用户复制资源到个人空间。

### 11.2 智能体对话

- 用户选择智能体发起对话。
- 智能体检索知识库。
- 智能体调用 Skill。
- 智能体调用 MCP Tool。
- 页面展示回答、引用来源和工具调用过程。
- 后台可查看完整调用日志。

### 11.3 流程编排

- 用户创建流程。
- 流程包含开始、智能体、Skill、条件、人工确认、输出、结束节点。
- 用户发布流程。
- 用户手动触发流程。
- 人工确认节点生成待办。
- 审核后流程继续执行。
- 用户查看流程实例和节点日志。

### 11.4 多智能体协同

- 用户配置主智能体和两个子智能体。
- 子智能体按顺序执行。
- 第二个子智能体接收第一个子智能体输出。
- 汇总智能体生成最终结果。
- 系统记录每个智能体的输入、输出和模型调用。

## 12. 关键风险与控制

| 风险 | 控制措施 |
|---|---|
| MVP 范围过大 | 严格限制第一版节点、MCP 类型、Skill 类型和协同方式 |
| MCP 安全风险 | 默认私有，公用发布必须审核，高风险工具必须确认 |
| Web 智能体危及服务器 | Tool、MCP、脚本和文件处理必须进入沙箱，禁止直接在 API 服务或宿主机执行 |
| Tool 越权 | 每次调用同时校验用户、智能体、资源和工具权限 |
| Prompt 不可追溯 | 智能体发布时固化 Prompt 版本 |
| 公用资源质量差 | 发布前要求描述、Schema、调试记录和审核 |
| 流程执行不可排查 | workflow_instance 和 node_instance 全量记录输入输出 |
| 多智能体结果不可解释 | 每个协同步骤独立留痕 |
| 知识库回答不准确 | 必须返回引用来源，支持用户反馈 |

## 13. 第一版不做事项

- 商业化插件市场。
- 任意用户上传并运行不受控代码。
- Web 智能体直接访问服务器宿主文件系统、系统命令或内网敏感服务。
- 完整 BPMN 规范。
- 复杂并行网关和补偿事务。
- 多智能体自主无限扩展。
- 跨租户资源共享。
- 复杂计费和成本分摊。
- 桌面端深度控制本地系统。
